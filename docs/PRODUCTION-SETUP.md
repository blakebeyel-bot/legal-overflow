## 1. Supabase — magic-link redirect URL config

**Status:** ✅ done (magic-link sign-in verified on live site)

**Problem (resolved):** Clicking the magic-link email on the live site
redirected to `http://localhost:8888` because the Supabase project's
Site URL was set to localhost during initial development.

**Fix:**

1. Open Supabase → your project → **Authentication** → **URL Configuration**
2. **Site URL:**
   ```
   https://resplendent-lollipop-59d4c4.netlify.app
   ```
3. **Additional Redirect URLs** (one per row):
   ```
   https://resplendent-lollipop-59d4c4.netlify.app/**
   http://localhost:8888/**
   ```
   The `/**` wildcards let any path under that origin be a valid redirect
   target — avoids having to enumerate every page.
4. Click **Save**.
5. Request a fresh magic link on the live site to verify.

## 3. Netlify function logs — turn on failure notifications

**Status:** open

Netlify can email you when a function fails in production (off by default).

1. Netlify → **Site configuration** → **Notifications**
2. Add notification → **Deploy failed** (if not already on)
3. Add notification → **Function failed**

This catches silent bugs in `fanout-background` that wouldn't surface
in the UI (review row frozen in a non-terminal status).

## 5. Supabase — custom SMTP for magic-link emails

**Status:** open (blocking real usage)

Supabase's free-tier built-in SMTP caps magic-link emails at **2 per
hour per project**. Hit it constantly during development; any live
user trying to sign in after a teammate also tried will see "Email
rate exceeded."

**Fix (one-time, ~10 min):**

1. Pick a transactional-email provider (all have free tiers):
   - **Resend** — easiest DX, 3k/mo free
   - **Postmark** — solid deliverability, 100/day free
   - **SendGrid** — 100/day free
2. Create an account, verify a sending domain (or start with their
   sandbox address for testing).
3. Grab the SMTP credentials (host, port 587, username, password).
4. Supabase → **Authentication** → **SMTP Settings** → toggle
   "Enable Custom SMTP" → paste credentials → **Save**.
5. Also set a friendly "Sender Name" / "Sender Address" on that
   page so emails arrive from `noreply@yourdomain.com` instead of
   `noreply@mail.supabase.io`.

**Current state (partial — Resend SMTP wired but sandbox-locked):**

Resend is configured and relaying via Supabase, but the test sender
`onboarding@resend.dev` can only deliver emails to the address that
owns the Resend account. Any other recipient fails with
"Error sending confirmation email."

To unlock full delivery:

1. Buy / repurpose a domain you control (e.g., `legaloverflow.com`).
2. Resend → **Domains** → **Add Domain** → paste the domain.
3. Copy the DNS records Resend shows (SPF TXT, DKIM TXT, one more).
4. Paste those into your domain registrar's DNS settings.
5. Wait for verification (5–30 min). Resend will email when ready.
6. Supabase → **Authentication** → **SMTP Settings** → change the
   **Sender email** from `onboarding@resend.dev` to something like
   `noreply@yourdomain.com`. Save.
7. Any recipient now works.

**Interim workaround:** Supabase → **Authentication** → **Rate
Limits** → bump the OTP limit from 2/hr to 30/hr. Still uses the
free-tier SMTP (capped at 30/hr max), but buys you headroom for
solo testing.
